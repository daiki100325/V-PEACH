# V-PEACH — 売上データ自動取り込み（Airメイト / Airレジ）検討資料

> **位置づけ**：Phase 7-4 にて実装完了。月次入力モードにCSV インポートサブモードを追加し、Airメイト + Airレジ CSV から提供売上・物販売上を自動算出する仕組みが稼動している。本ドキュメントは設計の背景・仕様・未解決事項の記録として保持。

---

## 1. 背景と目的

### 1.1 現状の運用

- **実装前**：月次入力モード（`InputApp.vue`）で毎月オーナーが店舗ごとに提供売上・物販売上・人件費の3項目を手入力していた。
- **Phase 7-4 以降**：CSV インポートモードが月次入力のデフォルトになり、提供売上・物販売上は Airメイト + Airレジ CSV から自動算出。手入力は人件費のみ。

### 1.2 目的

**毎月末に CSV をアップロードするだけで、各店舗の提供売上・物販売上が事業月度ベースで正しく `pe_monthly_records` に格納される**仕組みを構築する。

### 1.3 制約（事業月度の特殊性）

事業月度は暦月と一致せず、**基本的に前月の最終盤の日付を含む**（棚卸し日基準）。例：

| 店舗   | 事業月度       | 期間                       |
| ---- | ---------- | ------------------------ |
| 馬場本店 | 2026年3月度   | 2026-02-26 〜 2026-03-22  |
| 馬場本店 | 2026年1月度   | 2025-12-25 〜 2026-01-29  |

事業月度の境界は V-MINT 2.0 の `cost_reports.start_date` / `end_date` で既に管理されており、V-PEACH からも参照可能。

**この性質が引き起こす問題**：1ヶ月分のインポートを行うのに、Airレジ側は「前月の最終盤 + 当月分」の **2ヶ月分の CSV** が必要になる。単純に運用すると毎月手順が倍になる → §2.2 で対策を述べる。

## 2. データソースの整理

V-PEACH に取り込む元データは **2系統 × 3店舗**。当月インポートで実際にユーザーがアップする CSV は **2ファイル/店舗 × 3店舗 = 6ファイル**（前月分の Airレジ CSV は DB キャッシュから自動取得）。

### 2.1 商品別売上CSV（**Airメイト**から取得）

- 取得元：**Airメイト**（Airレジではない点に注意）
- 期間指定：任意期間を指定してエクスポート可能 → **事業月度ピッタリで指定できる**（前月最終盤を含む期間でOK）
- 形式：商品単位ではなく **カテゴリー集計** で出力される
- カテゴリーマッピング（1対1対応）：

| カテゴリー名 | V-PEACH での意味 | カラム |
|------------|------------------|--------|
| `シーシャ` | 提供売上（割引前） | カテゴリー別売上 |
| `物販` | 物販売上 | カテゴリー別売上 |

- **欠点**：割引額が反映されない（割引前の総額が出る）
- このCSVは月次集計値しか出ないため、**毎月「事業月度ピッタリ」で指定して取り直すしかない**（DB キャッシュ化の余地なし）

### 2.2 日別売上CSV（**Airレジ**から取得） + **日次キャッシュ戦略**

- 取得元：**Airレジ**
- 期間指定：任意期間を指定してダウンロード可能
- 形式：1日1レコード。日次の売上額・**割引額**・客数などを含む
- 役割：**割引総額の算出根拠**として使う
- **前提運用**：弊社では **割引は提供売上にしか付かない** → 割引総額は提供売上から差し引けばよい（物販への按分は不要）

#### 2.2.1 単純運用の課題

事業月度が前月最終盤を含むため、3月度（例：2/26〜3/22）の割引を集計するには 2月末の日次データも必要。素直にやると毎月「前月分 + 当月分」の Airレジ CSV を2回アップロードすることになり、手順が倍。

#### 2.2.2 採用方針：日次キャッシュテーブル

新規テーブル `pe_daily_sales_cache` に、アップロードされた Airレジ日別データを **店舗 × 日付** 単位で永続化する。

```text
毎月の運用：
  1. ユーザーは「当月暦月全体」の Airレジ CSV を1店舗1ファイルだけアップロード
     （例：3月度処理時 → 「2026-03-01 〜 2026-03-31」の CSV）
  2. システムは CSV を pe_daily_sales_cache に upsert
  3. 事業月度範囲（例: 2/26〜3/22）の割引集計は、
     - 前月最終盤（2/26〜2/28）→ 前月インポート時に保存済みのキャッシュから取得
     - 当月分（3/1〜3/22）→ 今回アップロードした CSV から取得
  4. インポート末尾で、当月度 start_date より古いレコードを自動削除（§5.3 参照）
```

→ **毎月アップロードする Airレジ CSV は1店舗1ファイルだけで済む**（手順倍化を回避）。

#### 2.2.3 初回データ投入（2025年12月分のみ）

本アプリは **2026年1月分から記録を開始** する想定のため、最初の自動運用月（2026年1月度）の前月最終盤データだけ事前に揃えておけばよい。

→ **2025年12月分のみ、Supabase 上で SQL を直接叩いて `pe_daily_sales_cache` にインポート**する。これ以降は毎月の通常フローで自然に前月データが残るため、UI 側に「初回専用フロー」を実装する必要はない。

### 2.3 ファイル配布の単位

- 3店舗（馬場本店・中野店・馬場2号店）は **店舗ごとに別ファイル** で取得する
- 1回のインポート対象 = **2ファイル/店舗 × 3店舗 = 6ファイル**

## 3. 計算ロジック

事業月度 `[start_date, end_date]` を確定したうえで、店舗ごとに以下の手順で算出：

```text
[STEP A: 当月のAirレジ日別CSVを pe_daily_sales_cache に upsert]
  CSV の各行を { store_id, sale_date, discount_amount, ... } として upsert
  → 当月暦月の日次レコードが DB に揃う
  （前月最終盤のレコードは前月インポート時に保存済み）

[STEP B: 事業月度範囲の割引総額を集計（DBから）]
  SELECT SUM(discount_amount)
    FROM pe_daily_sales_cache
   WHERE store_id = :store_id
     AND sale_date BETWEEN :start_date AND :end_date
  → total_discount

[STEP C: Airメイト商品別売上CSV（事業月度ピッタリ）から割引前売上を取得]
  raw_service_sales      = カテゴリー「シーシャ」の合計
  raw_merchandise_sales  = カテゴリー「物販」の合計

[STEP D: 最終値の算出]
  service_sales      = raw_service_sales − total_discount   ← 真の提供売上
  merchandise_sales  = raw_merchandise_sales                ← 物販はそのまま

[STEP E: 古いキャッシュを削除]
  当月度の start_date より古い pe_daily_sales_cache レコードを店舗ごとに削除
  （§5.3 のロジック参照）
```

> 物販に割引が付かない運用が崩れた場合は、按分ロジック追加が必要（→ §7 Open Questions）。

## 4. UI/UX フロー案

月次入力モード（InputApp）に **「売上CSVインポート」サブモード** を追加する案を本命とする。

### 4.1 フロー全体

```text
Step 1: 期間選択（YYYY年MM月）
  ↓
  V-MINT 2.0 cost_reports から事業月度境界を自動取得・画面表示
  「2026-02-26 〜 2026-03-22」のように表示
  ↓
Step 2: 店舗ごとにCSV 2ファイルをアップロード
  ※ Airレジは「当月暦月全体」を指定。前月最終盤は DB キャッシュから自動取得
  [馬場本店]  商品別売上(Airメイト・事業月度ピッタリ).csv  +  日別売上(Airレジ・当月暦月).csv
  [中野店]    商品別売上(Airメイト・事業月度ピッタリ).csv  +  日別売上(Airレジ・当月暦月).csv
  [馬場2号店] 商品別売上(Airメイト・事業月度ピッタリ).csv  +  日別売上(Airレジ・当月暦月).csv
  ↓
Step 3: 計算結果のプレビュー
  店舗 | 提供売上（割引後）| 物販売上 | 割引総額 | 元の提供売上 | 前月キャッシュ参照日数
  ---------------------------------------------------------------------------------------
  馬場本店  | ¥2,150,000 | ¥150,000 | ¥50,000 | ¥2,200,000 | 3日（2/26〜2/28）
  中野店    | ¥2,800,000 | ¥200,000 | ¥40,000 | ¥2,840,000 | 4日
  馬場2号店 | ¥1,550,000 | ¥150,000 | ¥30,000 | ¥1,580,000 | 2日
  ↓
Step 4: 確認 → 保存
  - pe_monthly_records への upsert（3店舗）
  - pe_daily_sales_cache の古いレコードを自動削除
  - 既存レコードがあれば上書き確認ダイアログ
```

### 4.2 人件費の扱い

CSV では人件費は取れないため、Step 3 のプレビュー画面で **人件費入力欄を追加** し、売上と一括で upsert する（現行の月次入力フローと統合）。

### 4.3 エラー処理

| ケース | 挙動 |
|--------|------|
| CSV 形式不正（ヘッダー欠落・カテゴリー欠落） | 該当店舗のみエラー表示・先に進めない |
| Airメイト CSV の期間が事業月度と不一致 | 警告表示（ユーザーが期間指定を間違えた可能性）。続行可だが要確認 |
| Airレジ CSV が当月暦月をカバーしていない | 警告表示。事業月度 end_date まで含まれているかを最低条件にチェック |
| **前月最終盤のキャッシュが DB に存在しない** | 「前月分の日次データが見つかりません。前月インポート漏れの可能性あり。SQL で手動投入してください」エラー |
| 日別売上CSV の対象日が0件 | 割引総額0として扱い、警告表示 |
| V-MINT 2.0 に該当月の `cost_reports` がない | 棚卸し未完了 → インポート不可エラー |

## 5. 実装方針

### 5.1 技術スタック

- フロントエンド + Supabase（`pe_daily_sales_cache` のみ DB 経由）
  - パーサ：**自前実装**（papaparse は不採用。クォート対応・金額バックスラッシュ/円マーク/カンマ除去を含む）
  - エンコーディング：`TextDecoder('shift-jis')` でブラウザ内変換（実機確認済み）
  - 実装ファイル：`src/utils/csvImporter.js`
- パース結果は state に保持し、最終保存時に `pe_daily_sales_cache` upsert + `upsertMonthlyRecord` + 古いキャッシュ削除をトランザクション的に実行

### 5.2 V-MINT 連携

既存 `getCostReportDates(storeId, periodKey)`（`src/api.js`）が `cost_reports.start_date` / `end_date` を返している。これを Step 1 で叩いて期間境界を取得・表示。

### 5.3 DB スキーマ変更

#### 5.3.1 新規テーブル `pe_daily_sales_cache`（必須）

```sql
CREATE TABLE pe_daily_sales_cache (
  id              uuid PRIMARY KEY DEFAULT uuid_generate_v4(),
  store_id        bigint NOT NULL REFERENCES stores(id) ON DELETE CASCADE,
  sale_date       date NOT NULL,
  discount_amount numeric NOT NULL DEFAULT 0,   -- 正の数で保持（CSVは負値 -300 等だが ABS で正規化）
  -- 監査・将来拡張用に元データの主要カラムも保持
  gross_sales     numeric,         -- 日次の総売上（任意）
  customer_count  integer,         -- 客数（任意）
  raw_payload     jsonb,           -- CSV原文（万が一の再計算用、任意）
  imported_at     timestamptz NOT NULL DEFAULT now(),
  UNIQUE (store_id, sale_date)
);
CREATE INDEX idx_pe_daily_sales_cache_store_date
  ON pe_daily_sales_cache(store_id, sale_date);

-- RLS（既存テーブル群と同じ anon_all ポリシー）
ALTER TABLE pe_daily_sales_cache ENABLE ROW LEVEL SECURITY;
CREATE POLICY "anon_all" ON pe_daily_sales_cache FOR ALL TO anon USING (true) WITH CHECK (true);
```

**設計理由**：
- `pe_monthly_records` とは別テーブルにすることで責務を分離（月次集計値 vs 日次キャッシュ）
- `store_id bigint REFERENCES stores(id)` — V-MINT2.0 と共用の `stores` テーブル（`pe_stores` は存在しない）を参照
- `discount_amount` は **CSV では負値（-300 等）だが DB には正の数で保持**（パース時に `Math.abs()`）。`service_sales = raw_service_sales - SUM(discount_amount)` で素直に計算できる
- `UNIQUE (store_id, sale_date)` で同じ日付の重複を防ぎ、再インポート時は upsert で自然に上書き
- 削除ロジック（§5.3.3）もこのテーブルに閉じる
- 実装済みマイグレーション：`supabase/DB_MIGRATION_daily_sales_cache_20260518.sql`

#### 5.3.2 `pe_monthly_records` への監査列追加（任意・拡張案）

```sql
ALTER TABLE pe_monthly_records
  ADD COLUMN raw_service_sales numeric,   -- 割引前
  ADD COLUMN discount_total numeric,      -- 期間内割引総額
  ADD COLUMN imported_at timestamptz,     -- インポート日時
  ADD COLUMN imported_source text;        -- 'csv' / 'manual'
```

→ オーナーが後から「この月、いくら割引したっけ？」を遡れるメリット。最小版でスタートし、必要になったら追加でよい。

#### 5.3.3 古いキャッシュの自動削除ロジック

インポート処理の末尾（STEP E）で、店舗ごとに以下を実行：

```sql
-- 「今回処理した事業月度の start_date」より古いレコードを削除
DELETE FROM pe_daily_sales_cache
 WHERE store_id  = :store_id
   AND sale_date < :current_period_start_date;
```

例：3月度（start=2026-02-26）のインポート終了時、2026-02-25 以前を削除。

これにより、各店舗のキャッシュは常に **「現在処理中の事業月度の start_date 以降」** だけが残る → 3店舗 × 30日前後 = **常時 90行前後** に収まる。

### 5.4 既存手入力 UI の扱い

CSV インポート機能追加後も、**手入力モードは残す**。
- 過去月の修正
- CSV が出ない・壊れた等のフォールバック
- インポート画面と手入力画面を **タブ切替** にする（同じ Step 1/4 構造の上に乗せる）

## 7. Open Questions（オーナー確認待ち）

1. **物販割引の前提崩れ**：将来「物販にも割引を付ける」運用に変わる可能性は？ → ある場合は按分ロジック必須。
2. ~~CSV のエンコーディング・カラム名~~ ✅ 確定済み（§2.1 / §2.2 に反映）
   - **Airメイト**: Shift-JIS / カラム `順位,カテゴリー,出数ABC分析,出数構成比,出数合計,売上ABC分析,売上構成比,売上合計,粗利ABC分析,粗利構成比,粗利合計`。金額は `\1644700` 形式（バックスラッシュ + 数値、円マーク）。**期間情報は CSV 内に無く、ファイル名から取得が必要**
   - **Airレジ**: Shift-JIS / カラム `集計期間,売上,会計数,会計単価,客数,客単価,商品数,現金支払合計額,現金その他支払合計額,割引額`。集計期間は `YYYYMMDD`、**割引額は負値**（-300 等）、休業日（売上0日）はそもそも行が出ない
3. **重複インポートのルール**：同じ事業月度を再インポートしたら上書き or 警告のみで上書き不可？ → デフォルトは「警告 → 上書き確認ダイアログ」で進める想定。
4. **Airレジ CSV の期間指定運用**：「当月暦月全体」を毎月固定で出力してもらう運用で問題ないか？（例：月末締めで翌月3日までに出す等）
5. **棚卸し未完了月の扱い**：`cost_reports` がない月はインポート不可とするか、暦月で代替するか？ → §4.3 では「不可」前提。
6. **監査メタ列（§5.3.2）**：割引総額や元の提供売上を DB に保持する必要はあるか？
7. **`raw_payload` 列の要否**：再計算用に CSV 原文を JSONB で保持しておくか？容量は小さいが、不要なら削ってシンプルに。
8. **削除タイミングの粒度**：店舗ごとに事業月度境界が異なる場合の削除判定は店舗単位で良いか？（=「店舗 A の3月度 start」と「店舗 B の3月度 start」がズレても、店舗単位で独立に削除）

## 8. 想定リスク

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| Airメイト の CSV 仕様変更 | パース失敗 | カラム名検出を寛容に（複数候補から探す）。バージョンを CHANGELOG に記録 |
| Airレジ の日別売上 CSV に割引カラムがない店舗・期間 | 割引 0 として誤集計 | パース時に「割引カラムが見つからない」を明示エラーに |
| カテゴリー名揺れ（「シーシャ」→「shisha」等） | カテゴリー検出失敗 | カテゴリーマッピングを設定値化（`pe_company_settings` 等に持つ） |
| 棚卸し日が月をまたぐ／同日の重複 | 期間境界ズレ | V-MINT 側の `cost_reports` を真とするポリシーを明文化 |
| 既存手入力データとの突合 | 過去月をインポートしたら値が変わって PL 履歴が変動 | プレビュー段階で「既存値 vs 新値」を並べて表示 |
| **前月キャッシュ削除のタイミングミス**（インポート途中で失敗したのにキャッシュは消えた等） | 翌月のインポート時に前月最終盤データ欠落 → 割引集計が過小に | 削除は upsert・月次保存がすべて成功した後の最後に実行。エラー時はロールバック。手動 SQL リカバリ手順をドキュメント化 |
| **インポート順序ミス**（先月分をスキップしていきなり今月分） | 必要な前月キャッシュが DB にない | §4.3 で明示エラー化。誤って削除した場合は SQL で再投入 |
| `pe_daily_sales_cache` のスキーマ進化 | 過去レコードのマイグレーション必要 | `raw_payload` JSONB を持っていれば原文から再投入可能 |
