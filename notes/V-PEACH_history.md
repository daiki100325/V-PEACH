## 第1部：V-MINT 開発史（在庫管理ツールの誕生と進化）

### 1. 開発前夜 — 解決すべき課題

#### 1.1 現場の業務課題

シーシャ屋ばんびえんでは、月次棚卸し・在庫管理・店舗間の商品移動が長年にわたって手作業で行われていた。主な問題点は以下のとおりだった。

- 月末棚卸しの集計が**スプレッドシートへの手入力**で、転記ミスや抜け漏れが発生しやすかった
- 各拠点の棚卸し完了後、**各拠点のスプレッドシートから集計用スプレッドシートへの手作業による転記と目視確認**が必要で工数が膨大だった
- 在庫不足時の補充依頼と記録がメッセージアプリ頼りで、**LINEからスプレッドシートへの転記**が別途必要だった
- 担当者がスプレッドシートを**直接触らなくて済む UI** がなかった

#### 1.2 プロジェクトの出発点

上記課題を解消するため、ブラウザで動作する業務ポータルアプリの開発が開始された。初期コンセプトは「スプレッドシートはデータストレージとして維持しながら、業務担当者向けの入力UIをアプリとして提供する」というものだった。

### 2. V-MINT 無印 — フェーズ別開発史

#### 2.1 フェーズ 1: GAS HtmlService プロトタイプ

**状態: 完了**

最初のプロトタイプは **GAS（Google Apps Script）でHTMLを配信する**構成で実装された。

```
[GAS HtmlService（Play CDN TailwindCSS）]
    ↕ google.script.run（非同期）
[GAS スクリプト処理]
    ↕
[Google スプレッドシート × 4ファイル（店舗別）]
```

- フロントとバックエンドが同一のGASプロジェクトに同居
- 4店舗分のスプレッドシートがそれぞれ独立したファイル
- `google.script.run` 経由の非同期呼び出し
- TailwindCSSはPlay CDN（ランタイム生成）を使用

このフェーズで棚卸し入力・補充依頼を実装し、現場でのテスト・フィードバック収集を開始した。

---

#### 2.2 フェーズ 2: バックエンド統合と移動記録モード追加

**状態: 完了**

**スプレッドシートの1ファイル統合**

4店舗分を別々のファイルで管理していた構成を、**月ごとのシートを持つ1ファイル**に統合した。

- **効果1**: GASがファイルを開く処理が4回 → 1回になり、初期データ読み込みが大幅に短縮
- **効果2**: 各拠点スプレッドシートから集計用スプレッドシートへの転記作業が消滅
- **副次効果**: 全店舗の在庫をひとつのシートで横断参照できるようになり、クロス店舗機能の実装が容易になった

**移動記録モードの追加**

- **起票**: 移動元・移動先・数量を記録してスプレッドシートに書き込む
- **検品**: 受け取り側が数量を確認・修正し、完了チェックを付ける
- **廃棄**: 廃棄した数量をマイナス計上として記録

スプレッドシートの移動記録専用領域: `CW〜HL列`（4列1ブロック × 最大30ブロック = 月内30件まで）

`LockService` による排他制御を実装し、複数ユーザーの同時書き込みによるデータ破損を防止した。

#### 2.3 フェーズ 3: Vue3/Vite フロントエンド移行 — V-MINT リリース

**状態: 完了（V-MINT 1.0 として本番稼働）**

**移行の背景**

GAS HtmlServiceでのHTML配信には以下の限界があった。

- HTML肥大化により**初期表示が遅く**なり始めていた
- Play CDN（Tailwind のCDN版）では**ブラウザ描画中のカクつき**が発生していた
- コード量増加により `Code.gs` とHTMLが混在し、**開発・デバッグが困難**になっていた

**リリース版アーキテクチャ**

```
[Vue 3 SPA（Vite ビルド → Cloudflare Pages で静的配信）]
        │ HTTPS fetch（Content-Type: text/plain）
        ▼
[GAS Web アプリ URL（/exec）= 外部から叩ける JSON API]
        │
        ▼
[Google スプレッドシート（1ファイル・月別シート）]
```

GAS側は「HTMLを配信する」役割を完全に廃止し、**データの読み書きのみを行うAPIサーバー**として再構成された。

**技術選定**

| 項目 | 選択 | 理由 |
|------|------|------|
| フレームワーク | Vue 3（Options API） | 既存コードとの親和性が高く学習コストが最小 |
| ビルドツール | Vite 5 | HMRが速く設定がシンプル |
| スタイル | Tailwind CSS 3（ビルド時静的化） | Play CDNのカクつき問題を解消 |
| API通信 | fetch（`Content-Type: text/plain`） | GASのCORSプリフライト問題を回避 |
| ホスティング | Cloudflare Pages（無料枠） | 静的CDN配信 + pushだけでデプロイ完結 |

**リリース時の業務モード（4モード）**

| appMode | モード名 | 概要 |
|---------|---------|------|
| `null` | ポータル | 4モードへの起点メニュー |
| `'inventory'` | 棚卸し入力 | 月末棚卸しデータの多ステップ入力・送信 |
| `'request'` | 補充依頼 | 在庫不足の把握・依頼テキスト生成 |
| `'transfer'` | 移動記録 | 起票・検品・入荷・廃棄の4サブモード |
| `'stock'` | 在庫量確認 | 全フレーバーの現在在庫・前月消費の参照 |

### 3. V-MINT 1.0 — リリース後のアップデート

#### 3.1 棚卸し入力の完成度向上

| 改善 | 内容 |
|------|------|

| 前月消費マイナスチェック | Step4遷移時にバックエンドへドライランを実行（`checkNegativeConsumption`）し、Step4画面内のバナーで警告表示。送信自体は可能 |
| 送信確認モーダル | ネイティブ `confirm()` からカスタムモーダルに移行し、全環境で確実に表示されるようにした |

#### 3.2 補充依頼の在庫表示強化

- フレーバーカードに**依頼元店舗の現在在庫**を大きく表示（500g未満は赤・警告マーク）
- 他3拠点の在庫をコンパクトなカードで横並び表示（最大在庫拠点をグリーンハイライト）
- **「在庫ゼロ非表示」トグル**で全拠点0のフレーバーを一括非表示

#### 3.3 移動記録の機能拡充

- 起票・廃棄・入荷の数量入力画面に、移動元/実施店舗の在庫と他3拠点との比較を表示
- 起票ルートのヘッダーに移動元→先を常時表示するバッジを追加（スクロールしても見える）
- 従来は事務所のみだった入荷記録を、全拠点で記録できるよう拡張

#### 3.4 ナビゲーション・UXの全面改善

**ネイティブダイアログの廃止（`ConfirmDialog.vue`）**

モバイルChrome・PWA・一部WebView環境では `window.confirm()` がダイアログを表示せず即 `false` を返す問題があった。全ての確認ダイアログをアプリ内VueコンポーネントへVue `provide/inject` パターンで統一した。

```
App.vue が provide() で requestConfirm を提供
    ↓
各Appコンポーネントが inject して使用
    ↓
await this.requestConfirm('メッセージ') が true/false を返す Promise
```

**ブラウザバックと戻るボタンの統一**

`popstate` 発火時に `history.pushState` で現在状態を再積みし直してから確認ダイアログを表示する「インターセプト方式」を採用。これによりブラウザバックでも確認ダイアログが確実に表示されるようになった。

### 4. V-MINT 2.0 — メジャーアップデート

#### 4.1 移行の背景と動機

V-MINT 1.0（フェーズ3）の「Cloudflare Pages + GAS JSON API + Spreadsheets」構成は安定していたが、以下の構造的な制約が存在していた。

| 制約 | 内容 |
|------|------|
| 移動記録の月内上限 | スプレッドシートの列設計により30件/月の上限あり |
| GAS実行時間制限 | 大量データ処理時に30秒制限に近づくリスク |
| 高度な集計の困難さ | スプレッドシート構造では月次集計・履歴参照に限界があった |
| 管理者機能の欠如 | フレーバーマスタ管理・新銘柄登録などをアプリ内で行えなかった |
| スプレッドシートのメンテナンス性 | 月替り・年替り時にスプレッドシート側の編集・新規シートの作成などが必要だった |
| 原価計算機能の不在 | シーシャ提供実績・炭消費・ドリンク発注などを記録・分析する手段がなかった |

V-MINT 2.0では、これらの制約を根本から解消するため**バックエンドをGAS + SpreadsheetからSupabase（PostgreSQL）へ完全移行**した。

#### 4.2 V-MINT 2.0 アーキテクチャ

```
[ユーザー端末（ブラウザ）]
        │ HTTPS
        ▼
[Cloudflare Pages（本番ブランチ: v2）]
  └─ Vue 3 SPA（Vite + TailwindCSS 静的ビルド）
        │ Supabase JS Client
        ▼
[Supabase（PostgreSQL + RPC）]
  └─ 棚卸し・移動記録・在庫・原価・フレーバーマスタ
```

GAS依存は完全に廃止され、すべてのデータアクセスは `src/api.js` 経由でSupabase RPC / クエリに直結している。

#### 4.3 新規追加アプリ（V-MINT 2.0で追加）

V-MINT 1.0の4モードに加え、以下の2つが追加された。

| アプリ | ルートキー | 概要 |
|--------|-----------|------|
| ダッシュボード | `dashboard` | 在庫量 / 棚卸し結果 / 実質原価の月次確認 |
| 原価計算 | `cost` | シーシャ提供数・物販数・炭消費量・ドリンク発注額の月次入力と計算 |
| 管理者画面 | `admin` | フレーバーマスタの表示制御、新規ブランド・銘柄の登録 |

#### 4.4 主要データベーステーブル

| テーブル | 役割 |
|----------|------|
| `inventory_logs` | 棚卸しスナップショット。`period_key + store_id + flavor_id` でUPSERT |
| `transfer_logs` | 店舗間移動記録。`amend_transfer_record` RPCで修正をトランザクション処理 |
| `brands` | ブランドマスタ。`is_cost_group` / `cost_group_id` で原価集約グループを管理 |
| `flavors` | フレーバーマスタ。`is_active` で表示/非表示を制御 |
| `cost_reports` | 月次原価計算の本体（販売数・炭消費量等を保持。価格は持たず `cost_price_masters` を参照） |

#### 4.5 V-MINT 1.0 と V-MINT 2.0 の差分まとめ

| 観点 | V-MINT 1.0（無印） | V-MINT 2.0 |
|------|--------------------|------------|

| 移動記録修正 | 不可 | `amend_transfer_record` RPCによる修正サブモード追加 |

**管理者画面の機能拡張（2026-05-12 / 05-13）**

- `パッケージサイズ設定` サブモードを追加し、`brands.has_pkg_*` / `packages_configured` で棚卸し入力欄を動的制御

**単位原価のマスタ参照型への刷新（2026-05-23）**

`cost_reports` の価格6カラム（`price_*`）を DROP し、集計時に毎回 `cost_price_masters` を解決する構成へ移行。マスタ編集が過去月の集計結果にも即座に反映されるようになり、価格改定時の手動 UPDATE 運用を廃止。**V-PEACH 側も同一マスタを参照することで、両アプリ間の価格データの整合が自然に担保される構造になった**。

V-MINT 全テーブルに対し RLS（Row Level Security）を有効化（Option A: anon 全許可ポリシー）。URL 非公開・信頼ユーザー前提のため既存動作は変えずに Supabase 警告を解消。V-PEACH 側の `pe_*` テーブルも同タイミングで RLS 有効化。

**原価計算 UX の改善（2026-05-25 / 05-28）**

- シーシャ販売数の各項目に ①〜⑤ を付番し、シーシャ提供本数（①+③+④+⑤）・滞在時間にかかる来店人数（②+③+④）の算式を UI に表示

- ドリンク発注の追加・編集時に前月の `cost_reports.start_date`〜`end_date` と発注日を照合し、範囲内なら保存をブロック。誤月登録（例: 6月モードで 5/20 の発注を追加）を防止

## 第2部：Supabase 移行が V-PEACH の早期実現をもたらした理由

### 5. GAS 時代の「見えない天井」と、Supabase が取り除いたもの

V-MINT 1.0 の GAS + スプレッドシート構成では、**経営ダッシュボードの実現は構造的に不可能に近かった**。その理由は技術的な限界というより、データ構造そのものにあった。

| 問題 | GAS + スプレッドシート時代 | Supabase 移行後 |
|------|--------------------------|----------------|
| 月をまたぐ集計 | 月シートをまたいだ JOIN 相当の処理をGASで手書きする必要があり、30秒実行制限に容易に抵触 | PostgreSQL の通常クエリで即解決 |
| 複数店舗の横断分析 | 4ファイル（後に1ファイル）のシート構造を解析するコードが複雑化 | `store_id` による単純フィルタで横断参照 |
| 原価データの蓄積 | 月ごとのシートに数値を書き込むだけで、フレーバー単位の詳細な原価追跡は不可能 | `cost_reports` + `cost_price_masters` で価格改定も含めた完全な原価履歴を管理 |
| 財務アプリとのデータ共有 | スプレッドシートを別アプリが参照するには GAS API を別途公開する必要があり、認証・レート制限が問題に | 同一Supabase プロジェクト内の別テーブル群として直接 JOIN・参照可能 |
| 長期トレンド分析 | スプレッドシートの年次ファイルを横断する仕組みがなく、年次PLの自動生成は困難 | `period_key`（YYYYMM）の範囲クエリで任意期間を即座に集計 |

**V-MINT 2.0 が Supabase に移行した本質は「棚卸し・原価データを SQL で問い合わせ可能な状態にしたこと」だった**。この変化が、その後の V-PEACH 開発において決定的な意味を持つことになる。

### 6. V-PEACH 実現の決め手 — 同一 Supabase プロジェクトの共用

V-MINT 2.0 が完成した時点で、ばんびえんの**現場データはすべて適切に正規化されたRDB上に存在**していた。具体的には：

- `cost_reports` — 月次の提供数・炭消費量（フレーバー原価の計算に必要）
- `flavor_brand_sales` — ブランド別物販実績
- `drink_orders` — 月内のドリンク発注額（累積）

これらのテーブルを、V-PEACH は**新しいインフラを一切立ち上げることなく**参照できた。V-PEACH は同一の Supabase プロジェクトに `pe_` プレフィックスのテーブル群を追加するだけで、V-MINT の現場データと財務データを結合した PL 計算を実現した。

```
V-PEACH (pe_* テーブル群)         V-MINT 2.0 (既存テーブル群)
─────────────────────────         ────────────────────────────
pe_monthly_records (売上・人件費)
pe_store_settings (固定費)    ──→  cost_reports (フレーバー原価)
pe_company_settings (全社費)  ──→  flavor_brand_sales (物販原価)
pe_benchmarks_revisions       ──→  drink_orders (ドリンク原価)
         │
         └──── calcPL() で V-MINT 原価 + V-PEACH 財務データを統合 ────→ 完全な月次 PL
```

もし V-MINT が GAS + スプレッドシートのままだったなら、V-PEACH はスプレッドシート解析専用の GAS API を別途設計・実装し、その API の CORS 制限・実行時間上限・認証問題を乗り越えた上でフロントエンドから叩く必要があった。**Supabase 移行によって、この複雑なデータ連携レイヤーがゼロになった**のである。

## 第3部：V-PEACH 開発史

### 7. V-PEACH の誕生 — V-MINT 2.0 の棚卸し結果が「次の一手」を問いかけた

V-MINT 2.0 の原価計算モードが実運用に入ると、ダッシュボードに毎月「実質原価率」が現れるようになった。棚卸しをするたびに数字が更新される。しかし経営者の問いはそこで止まらなかった。

> 「原価率がわかった。では、家賃・人件費・光熱費を全部合わせたら、今月の手残りはいくらだったのか？」

V-MINT は「現場の実態を数値化するツール」であり、財務 PL を出すことは設計範囲外だった。そのギャップを埋めるために生まれたのが **V-PEACH** である。

コンセプトは明確だった：

> **V-MINT 2.0 が棚卸しで明らかにした原価実績を起点として、売上・固定費・人件費と結合し、経営判断に使える月次・年次 PL ダッシュボードを完成させる**

> **進捗サマリー（2026-06-13）**: P1（DB: `stores` 4列追加・`pe_store_shift_rules`/`app_ui_settings` 新設）→ P2（在庫系 RPC を `jsonb_object_agg` 行ベース化・旧出力と差分ゼロ検証）→ P3（フロント動的化: `getStores()` 起点・stock 辞書アクセス・色/名前マップ・`shiftImporter` を `pe_store_shift_rules` 参照に）→ P4（V-PEACH `SettingsApp` に店舗管理 GUI＋3ステップ追加ウィザード・`create_store_atomic` RPC）→ P5（休止トグル両アプリ連動・閉店翌月以降の集計除外）をすべて完了。統合 e2e 全手順 PASS・go-live 直前コードレビューもブロッカーなしで通過。残は P6（保険ブランチ v3/v2 プレビュー目視 → 本番 `v3→v2`/`v2→main` マージ → スモーク）と P7（旧ピボット RPC 撤去）。

V-MINT 2.0・V-PEACH はともに、店舗（`baba_main` / `nakano` / `baba_2nd` / `office`）が **DB・RPC・フロントの 3 層にハードコード**されている。そのため新店舗オープン時は Supabase RPC のピボット定義からフロントの店舗キー直アクセスまで広範囲の改修が必要で、店舗事業のスケールに追従できない。Phase 13 では、店舗の増減を **管理者が GUI から完結**して扱える構造へ移行する。

**背景となった課題**
- RPC（`rpc.sql`）が `case when store_key = 'baba_main' ...` で**店舗別カラムにピボット**しており、店舗追加のたび SQL 手書きが必要
- フロントは店舗キーが**オブジェクトのプロパティ名**（`item.stock.baba_main` 等）・色／名前マップ・`shiftImporter` の馬場2号店遅番補正として散在
- V-MINT と V-PEACH が**別々に店舗リストをハードコード**しており、二重保守になっていた

**改修の柱（確定設計）**
- **DB**: `stores` に `is_active` / `display_order` / `store_type` / `closed_at` を追加（additive）。シフト枠時間ルール `pe_store_shift_rules`（店舗×シフト種別×日種別で 6h/7.5h・改定履歴付き）と、両アプリ共有 UI 状態 `app_ui_settings` を新設
- **RPC**: ピボットを `jsonb_object_agg` の**行ベース**に再設計し、店舗数非依存に（以後の店舗追加で RPC 改修ゼロ）
- **フロント**: `getStores()` 起点で店舗リスト・stock 辞書アクセス・色／名前マップを両アプリで動的化。`office` 特例は `store_type` 分岐へ
- **GUI**: 店舗マスタ管理を **V-PEACH `SettingsApp` に集約**（single source of truth）。`stores` は共用テーブルのため、V-PEACH で 1 回登録すれば V-MINT にも自動反映され、**二重設定が解消**される。追加ウィザードは固定費・シフトルールまで必須入力の**一発確定**

**運用方針**
- 閉店は**論理削除（休止フラグ）**ベース。過去データは保持し、決算用に「休止店舗も表示」トグル（全社一括・DB 共有）で閲覧可能
- 大規模改修につき、本番ユーザーへの即時展開を避ける**保険ブランチ**で進める: V-MINT は `v2`(本番) と別に **`v3`**、V-PEACH は `main`(本番) と別に **`v2`** に subtree push（Cloudflare ではプレビュー扱い）。全テスト完了後に本番へマージし、最後にレガシー（旧ピボット RPC 等）を撤去する Phase を設ける

> この計画は「現場ツール（V-MINT）と経営ダッシュボード（V-PEACH）が同一 Supabase を共用する」という本システムの強みを、**多店舗展開というスケール要件にそのまま活かす**ものである。店舗が増えても 1 か所の登録で現場・経営の両アプリが追従する構造を目指す。

## 第4部：経営ダッシュボードとしての意義

### 10. V-MINT + V-PEACH が実現した「現場から経営へ」のデータフロー

V-MINT 無印〜V-MINT 2.0〜V-PEACH の一連の開発は、単なる機能追加の積み重ねではなかった。それは**現場の手作業をデータ化し、そのデータを経営判断の根拠へと昇華させる構造的な変革**だった。

```
【現場スタッフ】
    │ V-MINT で棚卸し・移動記録・補充依頼
    ▼
【V-MINT 2.0 データベース（Supabase）】
    │ 月次原価計算（原価計算モード）
    ↓ cost_reports / flavor_brand_sales / drink_orders
【V-PEACH calcPL()】
    │ + pe_monthly_records（売上・人件費）
    │ + pe_store_settings（家賃・固定費）
    │ + pe_company_settings（役員報酬・借入返済）
    ▼
【経営ダッシュボード（PLモード）】
    ├── 月次・3ヶ月平均・年次 PL
    ├── FLR 比（Food/Labor/Rent コスト構造）
    ├── Health Check（目標値との乖離を赤/緑で可視化）
    └── トレンドチャート（全指標の時系列推移）
    ▼
【経営者の意思決定】
    ├── 人件費比率が高い月を特定 → シフト見直し
    ├── F比（原価率）が上昇傾向 → 仕入れ価格交渉
    ├── R比（家賃比率）が全社的に高い → 店舗規模・事業構造の見直し
    └── 年次PLで純現金収支を確認 → 投資計画・借入返済計画の策定
```

### 11. 「FLR 比」が経営者に与えた新しい視座

飲食業・シーシャ業の経営健全性は、売上規模より**コスト構造の比率**で測られる。V-PEACH が常時表示する FLR 比は、その洞察を即座に提供する。

| 指標 | 意味 | 参照タイミング |
|------|------|---------------|
| **F比（Food Cost Ratio）** | 売上に占める原料原価。原価率管理・仕入れ価格交渉の根拠 | 毎月の棚卸し後すぐ |
| **L比（Labor Cost Ratio）** | 売上に占める人件費。シフト設計・スタッフ配置の是正指標 | 月次入力完了後 |
| **R比（Rent Ratio）** | 売上に占める家賃。店舗移転・多店舗展開の判断に影響 | 年次比較・店舗比較時 |

V-MINT が棚卸しデータを正確に蓄積し続けているからこそ、F比は毎月リアルタイムで更新される。棚卸しの精度が、経営判断の精度に直結する構造が完成した。

### 12. 設定改定履歴という「タイムマシン」

家賃・決済手数料率・役員報酬・ベンチマーク目標値——これらは変わらないように見えて、実際には変化する。V-PEACH は `effective_from`（YYYYMM）を持つ改定履歴テーブルで全設定値を管理しており、**過去の任意の月を現在と同じ精度で再現できる**。

この設計により：
- 家賃改定や人件費体系変更があっても、過去月の PL は当時の条件で正確に計算される
- 「あの月はなぜ利益が出なかったのか」という遡及分析が常に可能
- 年次トレンドが「条件を変えて再計算した仮想値」ではなく、実際のコスト構造に基づいた真の実績として積み上がる

### 14. 本プロジェクトが示したこと

このプロジェクトが示した最大の教訓は、**「現場ツールとしての品質が、経営ツールの精度を決める」**という事実である。

V-MINT が毎月正確な棚卸しデータを蓄積し続けたからこそ、V-PEACH の原価数値は信頼できる。担当者が V-MINT で移動記録を丁寧につけ続けたからこそ、在庫量の実態が把握できる。その実態が V-PEACH の F比に反映され、仕入れ交渉の根拠になる。

スプレッドシートから始まり、GAS プロトタイプを経て、Vue3 SPA へ移行し、Supabase によってデータを解放し、そして経営ダッシュボードへと昇華した——この旅路は、**現場の課題を一つひとつ誠実に解決してきた結果として、経営の可視化が自然に手に入った**という物語である。

#### 異例の開発スピードを支えた 4 つの要因

V-MINT プロトタイプが走り始めた 2026 年 2 月 15 日から、V-PEACH がほぼ完成する 2026 年 5 月 25 日まで、わずか約 **3 ヶ月半**。V-PEACH に絞れば、開発決定（5/14）からほぼ完成（5/25）まで **わずか 11 日**で機能が積み上がった。一般的な業務システム開発では考えにくいこのスピードは、偶然の産物ではなく、以下の 4 つの要因が重なった結果だった。

**① 実務者が開発者を兼ねる構造**

開発者自身がばんびえん馬場2号店の棚卸し担当者でもあった。仕様を外部から「要件として聞き出す」必要がなく、毎月の実際の棚卸しオペレーションの中で何が痛みであり、何が必要十分かを身体知として持っていた。現場検証と開発サイクルが同一人物の中で完結するため、フィードバックループが極限まで短縮された。

**② 社長直轄の裁量権**

開発者は社長直轄のポジションにあり、機能追加・DB設計変更・業務フロー変更などの意思決定を即座に下せる立場にあった。外部ベンダーへの発注・稟議・仕様確定会議といった通常のシステム開発が必要とする「調整コスト」がゼロだった。

**③ AI 駆動の開発環境**

**Antigravity**・**Cursor**・**Claude Code** など AI ツールを全面的に活用し、定型実装・DB マイグレーション・バグ修正・ドキュメント生成を劇的に加速した。人間が担う領域を「何を作るか・なぜ作るか」の意思決定と動作確認・品質判断に集中させることで、少人数開発の限界を大幅に超えた生産性を実現した。

**④ 戦略コンサル出身の知識基盤**

開発者は IT 系ファームの戦略コンサルタント出身であり、企業の経営判断プロセス・KPI 設計・財務構造への知見を持っていた。また要件定義・アーキテクチャ設計・ユーザーストーリー整理といったシステム開発上流工程の手順を一通り習得していたため、「何を作るべきか」を正確に言語化し、開発サイクルを迷いなく回すことができた。経営ダッシュボードが「FLR 比」という飲食業の普遍的指標を軸に設計され、かつ「原価だけ」でなく売上・固定費・人件費までの完全な PL として完成したのも、この背景があってこそである。

#### リリースは終点ではなく、継続的改善の起点だった

V-PEACH の正式リリース（2026-06-01）は到達点ではなく、改善サイクルの一通過点に過ぎなかった。現場ツール V-MINT 2.0 はリリース後も管理者機能・価格マスタ・原価計算 UX の改善を継続し、経営ツール V-PEACH もリリース直後から次の課題——**事業のスケール（出店・閉店）への追従**——に着手した。それが Phase 13「多店舗スケール対応」（§8-2）である。

リリース時点の両アプリは、店舗（馬場本店・中野店・馬場2号店・事務所）が DB・RPC・フロントの 3 層にハードコードされており、新店舗を増やすたびに広範囲の改修が必要だった。Phase 13 ではこの構造を解き、店舗マスタの増減を**管理者が GUI から 1 か所登録するだけ**で現場・経営の両アプリに反映される形へ作り替えた（RPC の行ベース化・フロントの辞書アクセス化・V-PEACH への店舗マスタ集約）。これは単なる機能追加ではなく、**「店舗が増えても破綻しない構造」への作り替え**であり、事業の成長そのものをシステムが先回りして支える設計思想の表れである。

この継続性が示すのは、これらのツール群が「一度作って終わりの完成品」ではなく、**事業の変化に合わせて育ち続ける生き物**だという事実である。現場の痛みを起点に生まれ（V-MINT）、その実態を経営判断へ昇華し（V-PEACH）、さらに事業のスケールに構造から追従していく（Phase 13）——この絶え間ない往復こそが、本プロジェクトが「作って終わり」のシステム開発と一線を画す本質的な価値である。
